# 草稿：既有GPU生态训练软件针对第三方Device的低侵入式Kernel集成

## 前言

既有GPU生态训练软件的定义可能有点广泛，本文就暂定围绕 transformers, verl, VeOmni 这几个我实际使用过的软件去说明。

在训练软件中做第三方device的亲和加速，方法也很多。但其中最立竿见影的，也是最常用手段是将 torch_npu 的一些亲和 API 直接以 patch 的形式合入原仓。

这也是目前最主流的一种方法。

你会在上述几个软件中各自找到一份名为 npu_kernel.py 的文件。

25年参与了一些类似工作，发现了几个问题：
- 这些训练软件构建时本就没有考虑第三方 device 接入的情况。当我们参与进来做第三方 device 训练加速时，针对这些软件必须考虑改变其原有结构，包括但不限于更改模型接口入参、新增单独承载 kernel 的文件等。这会为第一个 PR 的合入带来巨大的沟通成本。
- 同样地，因为上述原因，第一个 kernel 的合入需要花费的时间成本非常大。
- 在很长的一段时间内，我的工作重心都在处理这些 patch 带来的维护问题。因为训练软件本身就在持续发版，一些API的名称改变是常有的事，patch 自然失效。
- 每出现一个新的训练生态库，需要将之前做过的工作完全重复做一遍，无法实现复用。

我将上述几个问题的出现原因都定义为：这类方法的侵入性过高。

所以我一直在致力于找到一种更具低侵入性的 Kernel 集成方法。本文是24年10月-25年2月对此问题解决方案的阶段性总结。


## 关于低侵入式集成概念的进一步澄清

为了解释低侵入式集成，我先给GPU生态训练软件做一个很主观的定义，只要是使用 torch + CUDA + GPU 构建的训练软件，就算在这个生态内。

在这个简单定义下论述时，会发现 CUDA + GPU 第三方 device 都已经有硬性定义，我们能做的很有限。那么实际上所谓的侵入性修改的高低就看你对 torch 使用这层修改了多少。修改的多就是侵入式高，修改的少就是侵入式低。

那么之前的方案因为需要引入 torch_npu api，patch时与 torch 是替代关系，自然存在很高的侵入性。

注意，这里不是说使用了 torch_npu api 所以侵入性高，是只有形成替代关系时才会。区别这点很重要。

所以，解决的思路自然就有两个：
- 思路一：建立一种外部加载工具 + 单独算子库的机制。外部加载工具是多种 device 通用并共同承认权威性的。单独的算子库不同的 device 随便去折腾，只要能满足加载工具的输入格式就能使用。
- 思路二：跳出 torch 的掣肘。torch 和 torch_npu 的关系已经固化，kernel 这层需要新的构建方法，而新的构建方法是同时原生支持多种 device 并拿到的接口就是最佳性能。注意是原生支持，torch 目前不是原生支持，或者说原生达不到最佳性能。


## 实现一：huggingface/kernels 的原生支持工作

kernels 允许 transformers 库（理论上所有 Python 模型库都可以）直接从 HuggingFace-Hub 动态加载计算内核。

项目地址是 https://github.com/huggingface/kernels

kernels 符合上述思路一，由 huggingface 保证其权威性，单独的算子库由公开的 hub 承载，任何用户都可以独立维护一份。

它的低侵入性体现在使用 hub 加载 kernel 逻辑，不需要集成在训练软件中。


## 实现二：Liger-Kernel 的原生支持工作

Liger-Kernel 是纯 triton 构建的 kernel 库。提供了三层 level 的API，与transformers 高度适配，verl 和 VeOmni 都已支持。

项目地址是 https://github.com/linkedin/Liger-Kernel

Liger-Kernel 符合上述思路二，使用 triton 属于一种新的开发模式。同时支持多种 device 拿到最佳性能。

有些事与愿违的是 triton 之上又诞生了 triton-ascend。如果是在 triton 本身中构建第三方 device 的能力，那么 Liger-Kernel 的适配工作量应该非常小。

好在 Liger-Kernel 是对 triton 的又一层封装，只要对外提供的接口同名，且各自能拿到最佳性能，那么训练软件集成 Liger-Kernel 也可以说是有比较低的侵入性。

如果没看明白，举个例子。想象一下，如果有一层对 torch 和 torch_npu 的封装，且训练软件集成的是这一层，一样能低侵入性。
